# 深入大腦：`bopomofo_t9.schema.yaml` 完全解析

如果 `t9bopomo.yaml` 是鍵盤的「外觀與按鍵」，那麼 `bopomofo_t9.schema.yaml` 就是整個輸入法的「大腦（RIME 引擎設定）」。

這份文件會帶你逐一拆解我們專案中的 `bopomofo_t9.schema.yaml`，用最白話的方式告訴你：當你按下鍵盤時，大腦是如何把一堆數字或英文字母，變成正確的中文。

---

## 第一部分：基本資料 (schema)

```yaml
schema:
  schema_id: bopomofo_t9
  name: 注音九宮格
  version: "10.1"
```
這就像是這顆大腦的「身分證」。
- `schema_id`: 輸入法的內部代號。在設定預設輸入法時（例如 `default.custom.yaml`），就是填這個名字。
- `name`: 顯示給你看的名字，叫做「注音九宮格」。

---

## 第二部分：運作引擎的零件 (engine)

```yaml
engine:
  processors: ...
  segmentors: ...
  translators: ...
  filters: ...
```
這裡定義了大腦裡的「各個部門」。你只要知道這是一個處理流程就好：
1. **processors (處理器)**：接收你按下的按鍵。
2. **segmentors (切分器)**：把長長的一串按鍵，切成一個個的音節。
3. **translators (翻譯器)**：把音節拿去查字典，翻出中文字。
4. **filters (過濾器)**：把查出來的字排序、過濾（例如把最常用的字排前面）。

---

## 第三部分：拼寫規則 (speller) —— 魔法發生的地方

這是整個檔案**最核心、最重要**的地方。它決定了「你輸入的代碼，大腦會怎麼解讀」。

```yaml
speller:
  alphabet: "abcdefghijklmnopqrstuvwxyz1234567890ABCDEIJKLMNOPSRVXZ"
  initials: "abcdefghijklmnopqrstuvwxyz1234567890ABCDEIJKLMNOPSRVXZ"
```

### 補充知識：`alphabet` 與 `initials` 有什麼不同？
你可能會好奇，這兩行看起來一模一樣，那它們分別代表什麼意思呢？

1. **`alphabet` (合法字元集合)**：
   這告訴大腦「哪些按鍵是可以用來打字的」。只要是你設定在這裡面的英文字母或數字，大腦收到後就會開始查字典。
   如果使用者按了一個**不在**這串字裡面的按鍵（例如 `@` 或 `!`），大腦就會覺得：「這不是要用來拼音的符號」，然後直接把它當成標點符號印在螢幕上。

2. **`initials` (首字母集合)**：
   這告訴大腦「哪些按鍵可以作為一個『新字』的開頭」。
   當你連續快速打了一串代碼（例如 `246`），大腦需要把它「切開」成三個字。大腦會去看 `initials` 裡面的設定，來判斷哪裡可以當作新音節的起點。
   *舉個一般電腦鍵盤的例子：在標準拼音裡，`a` 可以是字母的開頭（如 `ai` 愛），但 `g` 通常是聲母，而 `ng` 放在字尾。如果 `g` 不在 `initials` 裡，大腦就不會把它當成新字的開頭。*
   
   **為什麼在我們的九宮格裡，這兩行長得一模一樣？**
   因為在 T9 九宮格輸入法裡，**每一個數字鍵（或長按送出的精確字母）都有可能是新字（新注音）的第一個音**。例如你按 `1`，它可能是 `ㄅ` (聲母，在開頭)，也可能是 `ㄚ` (韻母，有些字只有韻母，所以也是在開頭)。因此，我們把所有允許輸入的代碼都同時放進 `alphabet` 和 `initials` 裡，讓大腦擁有最大的彈性去猜字。

   **🤔 進階疑問：可是「ㄤ (O)」、「ㄥ (P)」這種鼻音結尾的韻母，怎麼能當開頭 (`initials`) 呢？**
   這是一個非常好的問題！如果是在「標準英文拼音」裡，`ng` 確實不會是開頭。
   **但在「注音」的世界裡，是有所謂「零聲母」的字的！**
   例如：
   - **昂 (ang)**：它的注音就只有單單一個 **「ㄤ」**。
   - **鞥 (eng)**：它的注音就只有單單一個 **「ㄥ」**。
   當你想打出「昂」這個字時，你按下的第一個鍵就是 `ㄤ`（對應代碼 `O` 或是數字鍵 `9`）。
   如果我們把 `O` 或 `P` 從 `initials` 裡面拿掉，大腦就會覺得：「這不合法，字不能從這個符號開始」，這樣你就永遠打不出「昂」或「鞥」這些字了。這也就是為什麼幾乎所有的注音符號代碼，都必須保留在 `initials` 裡面的原因！

   **🤔 追問：但「鞥 (ㄥ)」、「昂 (ㄤ)」開頭的字根本沒機會用到，太冷僻了吧？可以拿掉嗎？**
   從日常使用的角度來看，確實非常冷僻。但從「輸入法系統設計」的角度來看，這是一個「穩定性」與「字庫完整性」的權衡問題。
   在我們目前的 `bopomofo_t9.schema.yaml` 設定中，你可能會注意到這行註解：
   `# initial 去除 ㄥ(P) ㄤ(O) 因為他們開頭的字實在過於生僻`
   沒錯，原作者（或目前的設定）**的確選擇了把它們拿掉**！

   **這樣做的具體影響是什麼？**
   1. **打不出特定生僻字**：因為大腦不再承認 `O` (ㄤ) 和 `P` (ㄥ) 可以作為一個字的開頭，所以當你按下 `9` 想打出「昂 (ang -> O)」，或者按下 `v` 想打出「嗯/鞥 (eng -> P)」時，大腦會直接拒絕組合出這些單字。你在選字清單裡永遠不會看到這些單韻母字。
   2. **簡拼與連打的取捨**：雖然拿掉它們會犧牲極少數的生僻字，但好處是能降低大腦在「長句連打」時的猜測負擔。RIME 引擎在處理連續輸入時，高度依賴 `initials` 來判斷邊界。由於 `O` 和 `P` 同時也是許多其他注音的結尾（例如 ㄉㄤ、ㄉㄥ），把它們從「合法開頭」名單中剃除，可以減少系統因為把別人的尾巴誤認為新字的開頭，而導致斷詞錯誤的機率。
   
   總結來說，目前的設定認為「讓一般打字更不容易斷詞錯誤」的好處，大於「能打出極度生僻的昂/鞥」的好處。這是一種實務上的優化選擇！

### 最神奇的魔法：`algebra` (運算規則)
這裡是一連串的「替換與衍生規則」。它的作用是：**把字典裡的標準拼音/注音代碼，轉換成你在九宮格上會按的按鍵**。

這裡有兩種指令：
- **`xform` (變形/取代)**：把 A 強制變成 B。
- **`derive` (衍生/增加)**：原本的 A 留著，但也允許用 B 來代表 A。（這就是模糊輸入、九宮格的核心！）

我們來看看這五個步驟在幹嘛：

#### 步驟 1：避開聲調專用字母
```yaml
    # 1. 先搬移可能衝突的字母 (ㄑ->A, ㄒ->B)
    - xform/q/A/
    - xform/x/B/
```
**🤔 為什麼要這樣做？不是已經設定了 q, w, x, y 是聲調專用鍵了嗎？**
你可能覺得很奇怪：既然已經規定好 `q, w, x, y` 是打聲調用的，而且大小寫沒有重複，為什麼還要特別把 `q` 跟 `x` 轉換掉？

這是因為我們專案底層借用的是 **`terra_pinyin` (地球拼音)** 的字典檔！
在標準的拼音字典裡：
- 小寫的 `q` 原本被用來代表發音 **ㄑ** (例如 qi = 起)
- 小寫的 `x` 原本被用來代表發音 **ㄒ** (例如 xi = 希)

如果我們不先把字典裡的發音規則改掉，當大腦收到一個 `q` 時，它會非常困惑：「使用者現在打的這個 `q`，是指『一聲』，還是指聲母『ㄑ』？」

為了解決這個衝突，我們必須在第一步就強迫大腦把字典裡所有代表 `ㄑ` 的 `q`，全部換成大寫的 `A`；把所有代表 `ㄒ` 的 `x`，換成大寫的 `B`。
這樣一來，小寫的 `q` 和 `x` 就徹底空出來了，大腦就知道只要收到小寫 `q`，就 100% 是一聲，再也不會跟注音符號搞混！

#### 步驟 2：基本的注音與拼音轉換
```yaml
    # 2. 基本注音轉換 (🌟 修復組合鼻音韻母)
    - xform/yong/vP/    # ㄩㄥ (用)
    ...
    - xform/ai/I/       # ㄞ 變成大寫 I
    - xform/ei/J/       # ㄟ 變成大寫 J
```
這一段看起來很長，其實是因為我們的字典底層是「拼音（terra_pinyin）」。大腦在這邊透過一堆 `xform`，把原本的拼音字母組合，變成我們專案自訂的「注音單一字母代碼」。例如 `ai` 變成了大寫的 `I`，這樣後面的處理才會一致。

#### 步驟 3：聲調轉換
```yaml
    # 3. 聲調轉換 (放在最後，確保不會被前面的 xform 誤傷)
    - xform/1$/q/
    - xform/2$/w/
    ...
```
字典裡記錄的一聲是數字 `1`，二聲是數字 `2`。我們在這裡把它們轉成我們指定的 `q`、`w`、`x`、`y`。

#### 步驟 4：T9 模糊鍵映射（九宮格的靈魂）
```yaml
    # 4. T9 模糊鍵映射
    - derive/a/1/      # 允許按 1 來打出 a (ㄚ)
    - derive/I/2/      # 允許按 2 來打出 I (ㄞ)
    - derive/[MR]/3/   # 🌟 這裡的 [] 括號是什麼意思？
    ...
    - derive/^b/1/     # 允許按 1 來打出 b (ㄅ)
    - derive/^d/1/     # 允許按 1 來打出 d (ㄉ)
```
這是九宮格能運作的關鍵！
因為你在鍵盤上按了 `1`，大腦怎麼知道你要打 `ㄅ`、`ㄉ` 還是 `ㄚ`？
透過 `derive`，大腦在建立索引時會知道：「只要遇到發音有 `b`、`d` 或 `a` 的字，當使用者按下 `1` 時，通通都要抓出來給他選」。
這就是為什麼九宮格能做到「模糊輸入」。

**🤔 疑問：那個 `[MR]` 是什麼意思？為什麼要用中括號 `[]` 包起來？**
你非常眼尖！這是一種程式語法（正規表達式）的**「簡寫」**。
中括號 `[]` 的意思是**「這幾個裡面的任何一個都可以（只取其一）」**。

在我們的鍵盤上，`M` (代表 ㄢ) 和 `R` (代表 ㄦ) 這兩個注音，剛好都被分配到了數字鍵 `3`。
所以 `- derive/[MR]/3/` 這句話的意思是：「只要發音遇到 `M` **或者**遇到 `R`，都允許使用者按 `3` 來打出來。」

其實，這寫法完全等同於寫成兩行：
`- derive/M/3/`
`- derive/R/3/`
我們只是為了讓檔案看起來短一點、乾淨一點，所以把有著相同目標數字鍵的字母，用中括號 `[]` 全部包在一起而已！同樣的邏輯也用在後面的 `[Ni]` (轉成 6)、`[KL]` (轉成 8) 上面。

**🤔 追問：那為什麼不乾脆把同一個按鍵的所有字母都用 `[]` 寫在一起？而且有些字母前面還多了一個 `^` 符號（例如 `- derive/^b/1/`）？**
這牽涉到「聲母（開頭）」和「韻母（結尾）」的嚴格區分！

- **那個 `^` 是什麼意思？**
  在程式語法中，`^` 這個符號代表**「開頭的第一個字」**。
  `- derive/^b/1/` 的意思是：只有當 `b` **出現在這個注音的最開頭時**，才允許你按 `1` 來打它。
  為什麼要這麼嚴格？因為 `b` 代表的是聲母「ㄅ」，而聲母**只會、也只應該**出現在一個字的最前面！加上 `^` 可以防止引擎在處理複雜的拼音轉換時，不小心把出現在中間或尾巴的字母錯誤地判斷成「ㄅ」。

- **為什麼不全部混在一起寫成 `derive/^[bda]/1/` 或 `derive/[bda]/1/`？**
  因為「條件不同」！
  - 像 `b` (ㄅ) 和 `d` (ㄉ) 是聲母，必須限制在「開頭」，所以要加 `^`。你可以把它們合併寫成 `- derive/^[bd]/1/`。
  - 但是 `a` (ㄚ) 是韻母，它可能單獨出現（也是在開頭，如 ㄚ），也可能跟在聲母後面（不在開頭，如 ㄅㄚ -> ba）。如果我們對 `a` 加上了 `^` 的限制（要求它必須在開頭），那麼當你想打「ㄅㄚ(ba)」時，因為 `a` 已經不是第一個字母了，引擎就會拒絕讓你用 `1` 來打出 `a`，這樣你就打不出「爸」這個字了！

總結來說：
- **聲母 (ㄅㄉ等)**：必須加上 `^` 確保它只在開頭觸發。
- **韻母 (ㄚㄞ等)**：不能加 `^`，因為它隨時隨地都需要能被觸發。
因為這兩種字母的「發動條件」不一樣，所以我們不能偷懶把同一顆按鍵的聲母跟韻母全部塞進同一個 `[]` 裡面。

#### 步驟 5：結尾處理與模糊首碼
```yaml
    # 5. 結尾處理
    - derive/[qwxy]$//      # 允許你不打聲調也能出字
    - derive/^(.).+$/$1/    # 允許你只打每個字的第一個按鍵 (簡拼)
```
- `derive/[qwxy]$//`：這行讓大腦知道，就算使用者沒有輸入最後的聲調代碼 (`q, w, x, y`)，一樣可以讓他查到字。

**🌟 簡拼魔法大解密：`- derive/^(.).+$/$1/` 到底是什麼意思？**
這行看起來像火星文的程式碼，是讓你能打「簡拼」（例如打 `ㄐㄊ` 就能出 `今天`）的最大功臣！
我們把它拆開來看（這同樣是正規表達式）：
1. `^`：代表這個注音的**開頭**。
2. `(.)`：小括號的意思是**「把它抓起來存著」**。裡面的 `.` 代表**「任何一個字母」**。所以 `(.)` 的意思就是：**把開頭的第一個字母抓起來，當作 1 號記憶體**。
3. `.+`：`.` 是任何字母，`+` 是指「一個或多個」。合起來就是**「後面跟著的一長串字母」**。
4. `$`：代表這個注音的**結尾**。
5. `$1`：這是在斜線後面的部分，意思是**「把前面找到的那整串東西，替換成 1 號記憶體裡的東西」**。

**👉 白話文舉例：**
假設字典裡有一個字是「天 (ㄊㄧㄢ)」，經過前面的轉換，它的完整注音代碼變成了 `tM`。
當大腦讀到 `- derive/^(.).+$/$1/` 這行規則時：
1. 它看到 `tM`。
2. 它把開頭第一個字母 `t` 抓起來存進 `$1`。
3. 它看到後面還跟著其他的字母 `M`。
4. 最後，它把整串 `tM` 衍生出一個新的代碼：**`t`** (也就是 `$1`)。

大腦做完這件事後，它就學會了：「哦！原來如果使用者只打一個 `t`（對應按鍵 `4`），我也要把『天』這個字列入考慮選項中！」
因為每個字都會經過這個魔法，所以大腦會記住所有字的「第一個注音」。當你連續打 `2` 和 `4` 時，大腦就知道你在找「開頭是ㄐ(2)」加上「開頭是ㄊ(4)」的組合，於是成功拼出「今天」！

## 第四部分：翻譯器與顯示設定 (translator)

這裡設定了字典的來源，以及你在打字時，螢幕畫面上（氣泡或輸入區）會顯示什麼文字。

```yaml
translator:
  dictionary: terra_pinyin
  prism: bopomofo_t9_custom # 🌟 獨立的拼寫緩存檔案名稱
  enable_user_dict: true    # 啟用使用者自造詞（會記住你常打的字）
  enable_sentence: true     # 啟動自動造句（會根據上下文幫你選字）
```

**🤔 疑問：`prism: bopomofo_t9_custom` 這是什麼？為什麼需要它？**
這是一個非常重要的設定，它叫做「拼寫映射（稜鏡）」。
在 RIME 裡，`dictionary` (字典) 是記錄「字怎麼唸」（例如：天 = tian1），而 `algebra` (運算規則) 則是決定「鍵盤怎麼按」（例如：按 4 然後按 3）。

當你部署 RIME 時，大腦會拿著 `dictionary`，套用所有的 `algebra` 規則，產生一個厚厚的「鍵盤按鍵對應表（副檔名為 `.prism.bin`）」，這樣你打字時才能飛快地跑出字來。

- **問題來了**：我們的 `bopomofo_t9` 跟一般電腦的 `terra_pinyin` (地球拼音) 共用**同一本字典** (`terra_pinyin.dict.yaml`)！
- **解決方法**：如果我們不設定 `prism`，大腦就會把我們這堆充滿「九宮格數字」和「大小寫亂改」的奇怪 `algebra` 規則，直接覆蓋掉原本正常的地球拼音對應表！這會導致如果你同時有安裝標準拼音，你的標準拼音就會壞掉。
設定 `prism: bopomofo_t9_custom`，就是告訴大腦：「請把我這套九宮格專用的按鍵對應表，**另外存成一個叫 `bopomofo_t9_custom.prism.bin` 的檔案**。」這樣就能確保我們的 T9 魔改規則，與別人共用同一本字典時能和平共處，互不干擾！

### 你螢幕上看到的注音怎麼來的：`preedit_format`
當你在鍵盤上打字時（例如按了 `1`，送出代碼 `b`），如果螢幕上顯示 `b` 或 `1`，那看起來很像在打亂碼。我們希望螢幕上顯示真正的注音符號。

```yaml
  preedit_format:
    - xform/b/ㄅ/
    - xform/p/ㄆ/
    ...
    - xform/1/ㄅ/
    - xform/2/ㄍ/
```
這裡的作用就是「視覺替換」。它**不會**影響查字典的結果，它只負責把你按下的字母或數字，在螢幕上「翻譯」成你看得懂的注音符號。例如大腦收到 `b`，它就在螢幕上畫出 `ㄅ`。如果是九宮格模糊輸入傳來的 `1`，它會預設顯示該按鍵的第一個注音（例如 `ㄅ`）。

---

## 第五部分：辨識器 (recognizer) —— 什麼是打字？什麼是符號？

在檔案的最下方，你會看到這樣一個區塊：
```yaml
recognizer:
  import_preset: default
  patterns:
    abc: "^[a-zA-Z0-9]+$"
```

**🤔 為什麼需要這個設定？**
RIME 引擎非常聰明，但它一開始有個預設的刻板印象：**「只有打小寫英文字母，才是要拼音查字典的；如果你打數字或是大寫字母，你應該只是想直接打出那個數字或大寫字母吧？」**

想像一下，如果你用一般的電腦鍵盤打字，按 `1` 就是要打出 `1`，你不會希望按了 `1` 跑出注音的 `ㄅ` 給你選。

但在我們的 **T9 九宮格鍵盤** 裡，這個邏輯被打破了！我們按下 `1` 是為了要打 `ㄅ、ㄉ、ㄚ`，我們甚至在前面的設定裡用大寫字母 `A`、`I` 來代表某些注音。

這時候，我們就必須透過 `recognizer`（辨識器）來糾正大腦的觀念！
- `patterns` 底下的 `abc`，意思是「定義什麼叫做正常的拼音輸入」。
- `^[a-zA-Z0-9]+$` 這是一段正規表達式，它強硬地告訴大腦：**「從現在開始，不管是小寫字母 (a-z)、大寫字母 (A-Z)，甚至是數字 (0-9)，只要使用者按了它們，通通都給我當作是要查字典的拼音代碼！」**

有了這行設定，當你按下數字鍵 `1` 時，大腦才不會直接在螢幕上印出 `1`，而是乖乖地把它送去前面的 `speller` (拼寫規則) 進行轉換與查字典！

## 總結

現在你知道 `bopomofo_t9.schema.yaml` 是怎麼運作的了：
1. 它先定義了字典裡各種發音該怎麼轉換（**步驟 1~3**）。
2. 它把那些發音綁定到了 `1~9` 的數字鍵上，實現了九宮格的模糊輸入（**步驟 4**）。
3. 它提供了一些方便的設定，像是可以不打聲調、可以打簡拼（**步驟 5**）。
4. 最後，它負責把你敲打的數字或字母代碼，美化成螢幕上的注音符號（**preedit_format**）。

所以，如果你想改變「某個注音要在哪個數字鍵上」，你要改的就是 **步驟 4** 的 `derive` 規則，以及下面 `preedit_format` 的視覺顯示規則！
