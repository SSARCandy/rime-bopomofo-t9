# 深入大腦：`rime.lua` (T9 智慧排序) 完全解析

這份文件詳細解釋了 `rime.lua` 腳本的運作原理。這個腳本要解決 T9 九宮格輸入法的兩個痛點：

1. **模糊輸入的雜訊**：一個數字鍵代表多個注音符號，長一點的輸入會冒出大量「碰巧符合按鍵、但根本不是你要的」的候選詞。
2. **長句斷詞難選**（v15 重點）：輸入一長串注音時，RIME 的整句猜測常常不是你要的，而正確的「第一個詞」又被壓在候選列深處，導致只能放棄、改成一個詞一個詞打。

## v15/v16 的核心思路：分段選詞 + 首音節提拔

輸入長句時，你真正需要的候選列長這樣：

1. **整句猜測**排最前面 —— 猜對時一鍵上屏，維持最快路徑。
2. **首音節最佳候選固定排第 2**（v16 新增）：只要第一個音節有打聲調，該音節的最佳單字就緊跟在整句猜測後面。這條規則可預期：「猜錯時，第 2 個候選永遠是第一個音節的字」。它專治單字虛詞——例如輸入「有比較厲害」時，「有」很容易被引擎跟「比較」的按鍵併在一起猜成別的詞，過去要翻好幾個候選才找得到單獨的「有」；現在它固定在第 2 位，閉著眼睛就能選。
3. 接著是**各種「斷詞長度」的最佳候選**：吃 6 碼的詞、吃 4 碼的詞、吃 2 碼的字……每種長度都有代表出現在前一頁。整句猜錯時，你只要從中挑出正確的**第一段**，RIME 就會對剩餘的輸入繼續出候選，一段一段選完即可。
4. **無效斷詞墊底**：如果某個候選詞結束的位置，後面緊跟著的是聲調鍵（q/w/x/y），代表這種切法會讓聲調變成「下一段的開頭」——但沒有任何音節是以聲調開頭的，這種斷法必錯，直接壓到最後。所以**只要你有打聲調，斷錯位置的雜訊就會被大量過濾掉**。

> v14 的「定錨提拔完美匹配」邏輯已移除：因為同一段輸入的所有候選字起點相同，「吃滿輸入的候選」本來就等於「覆蓋度最大的那一桶」，永遠排在最前面，該邏輯實際上不影響排序。v15 改用上述對分段選詞真正有幫助的策略。

## 腳本原始碼解析

### 1. 配置常數

```lua
local MAX_CANDS = 80        -- 參與排序的候選數上限
local TONE_SET = "[qwxy]"   -- 聲調鍵字元（q:ˉ w:ˊ x:ˇ y:ˋ）
```

- `MAX_CANDS = 80`：效能防護。RIME 在極端模糊的輸入下可能產生數百個候選，我們只對前 80 個做智慧排序，其餘照原順序附在後面，確保打字不卡頓也不漏字。
- `TONE_SET`：聲調鍵的字元集合，用來偵測「孤兒聲調」（見下文）。

### 2. 收集候選

```lua
    for cand in input:iter() do
        count = count + 1
        cands[count] = cand
        if count >= MAX_CANDS then break end
    end
```

先抓取前 80 個候選存進陣列。沒抓完的部分會在最後用第二次 `input:iter()` 原樣輸出（librime-lua 的候選流是接續的，不會重複）。

### 3. 分類：完整覆蓋 / 覆蓋長度分桶 / 孤兒聲調

```lua
        local c_end = cand._end or cand["end"] or 0
        local c_start = cand.start or 0
        local cov = c_end - c_start

        if c_end >= input_len then
            table.insert(full, cand)                -- 吃滿所有輸入
        elseif input_str:sub(c_end + 1, c_end + 1):match(TONE_SET) then
            table.insert(orphans, cand)             -- 斷在聲調鍵前：無效斷詞
        else
            table.insert(buckets[cov], cand)        -- 依覆蓋長度分桶
        end
```

每個候選字都帶有它「吃掉了輸入中哪一段」的資訊（`start` 到 `_end`）：

- **完整覆蓋 (`full`)**：吃滿目前所有輸入的候選——整句猜測、完整詞組。
- **孤兒聲調 (`orphans`)**：候選結束的下一個字元是聲調鍵。舉例：輸入 `266q463q`（今天），某個詞只吃到第 3 碼，剩下的輸入就會從 `q` 開始——不可能，必錯，墊底。
- **其餘候選**依覆蓋長度（吃了幾碼）分桶，桶內保持 RIME 原本的詞頻順序。

### 4. 首音節提拔（v16）

```lua
    local promoted = nil
    for i = seg_start + 1, input_len do
        if input_str:sub(i, i):match(TONE_SET) then
            local b = buckets[i - seg_start]
            if b and b[1] then
                promoted = table.remove(b, 1)
            end
            break
        end
    end
```

從候選共同起點往後找**第一個聲調鍵**——它標記了第一個完整音節的結束位置。該位置對應的覆蓋桶裡的第一個候選（＝該音節頻率最高的字）被抽出來，稍後緊跟在完整覆蓋候選之後輸出。沒打聲調時找不到錨點，不提拔，行為與 v15 相同。

### 5. 輸出：完整覆蓋 → 首音節 → 各長度輪流

```lua
    for _, cand in ipairs(full) do yield(cand) end
    if promoted then yield(promoted) end

    local round = 1
    local emitted = true
    while emitted do
        emitted = false
        for cov = max_cov, 0, -1 do
            local b = buckets[cov]
            local cand = b and b[round]
            if cand then
                yield(cand)
                emitted = true
            end
        end
        round = round + 1
    end
```

這是 v15 與 v14 最大的差異。v14 是「覆蓋度嚴格降冪」：先把所有吃 7 碼的輸出完，再輸出所有吃 6 碼的……結果長輸入時，正確的第一個詞（通常只吃 2～4 碼）被幾十個錯誤的長匹配壓到列表深處。

v15 改成**輪流輸出**：第一輪先給每種長度的「最佳」候選（由長到短各一個），第二輪給每種長度的「次佳」……這樣候選列前一頁就同時包含「最長的猜測」和「各種長度的首段選擇」，分段選詞才真正可行。

### 6. 保底輸出

```lua
    for _, cand in ipairs(orphans) do yield(cand) end
    for cand in input:iter() do yield(cand) end
```

無效斷詞墊底，然後把因 `MAX_CANDS` 限制沒處理到的候選依原順序補上，確保不遺漏任何字。

## 實際效果

以輸入 `266q463q`（今天，兩個音節都帶一聲）為例：

| 順位 | v14（覆蓋度降冪） | v15（分段選詞） |
|---|---|---|
| 1 | 整句猜測 | 整句猜測 |
| 2~N | 一堆吃 5~7 碼的錯誤長詞 | 最佳的 6 碼詞、**「今」(4碼)**、最佳單字…輪流 |
| 深處 | 「今」「金」等正確首段 | 錯誤的長匹配、孤兒聲調候選 |

## 總結

- **整句猜對時**：跟以前一樣，第一個候選直接上屏。
- **整句猜錯時**：正確的第一段就在前幾個候選裡，選下去、繼續選第二段，一路選完。
- **有打聲調時**：斷錯位置的候選（孤兒聲調）全部墊底，雜訊大幅減少——聲調鍵同時是「精確過濾」與「斷詞錨點」。
- **效能防護**：`MAX_CANDS` 確保極端模糊輸入下依然流暢。
